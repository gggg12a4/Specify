# 02 App 开发页面交互重构方案

> 入口：首页 App 卡片点击进入
> 核心理念：左侧知识，右侧主工作区（上指令、下能力）。降低新手认知门槛、减少视觉噪音、强化知识与指令与能力的联动、全面采用居中弹窗收敛深层配置。

---

## 一、 页面整体框架

采用现代化、类 IDE 的 **「两栏双屏沉浸工作流」**，极大释放主工作区的水平视觉宽度。

### 1.1 核心区域划分
1.  **顶部状态栏**：全局导航、模型选择与发布控制。
2.  **左侧侧边栏（知识库）**：资料树与沙盒文件系统。
3.  **右侧主工作区（指令+能力）**：垂直分割为两部分。
    *   **上方核心画布（指令区）**：占比约 60% 高度，纯净的系统提示词编辑区。
    *   **下方能力面板（工具区）**：占比约 40% 高度，分行展示并勾选能力。

### 1.2 线框图概览

## 二、 顶部导航栏与状态区

采用 **经典左中右三段式布局**。*注：当前页面仅为草稿编辑区，彻底取消【发布】与【版本历史】入口，强制用户前往调试页进行发版，落实“不测不发”机制。*

*   **左侧（上下文）**：`← 首页` | `App名称` (支持点击原位修改)。
*   **中间（全局模型选择）**：单一的下拉框（如 `Claude 3.5 Sonnet ▼`）。
    *   *重要约束：全局唯一大脑。主 App 以及其下挂载的所有子 Agent，强制使用此处选择的大模型，不支持子 Agent 独立选型，极大降低 C 端配置复杂度。*
*   **右侧（操作与状态）**：
    *   `[保存]`：提供明确的保存按钮供开发者手动保存配置。
    *   `[▶ 进入调试]`：高亮主按钮。基于当前草稿进入调试页面。所有的版本对比、发布上线均在此动作之后进行。

## 三、 主工作区联动 (知识 + 指令 + 能力)

### 3.1 左侧资源与沙盒面板 (文件系统)

左侧面板是资源与沙盒文件系统的可视化展示区，核心作用是帮助开发者理解和管理 AI 在运行时的**文件读写上下文与边界**。通过可视化目录树，明确平台的数据隔离机制：上半部分代表开发者灌输的“公共先验知识”，下半部分代表使用者在交互中产生的“私有动态数据”。

**1. 📁 App 资料库 (`shared/` 目录)**
针对**开发者**层面的全局知识库设定区域。
*   **核心定位**：共享的基准知识库（如业务规则、风格指南、样例数据）。在此存放的文件，所有使用该 App 的最终使用者在交互时，AI 都能读取。
*   **交互管理**：区块顶部提供 `[+ 新建文件]` 与 `[+ 上传资料]` 按钮。支持批量拖拽。
*   **路径引用**：在文件项（如 `规范.md`）悬停时，显示 `[📋复制路径]` 操作。开发者可无缝粘贴到右侧指令中（如“请遵守 `shared/规范.md`”）。
*   **文件操作**：悬停时提供 `[✎ 重命名]`、`[🗑 删除]`。

**2. 📂 用户空间 (独立隔离沙盒)**
展示针对**最终使用者（End User）**在发布后的运行时标准目录规范。Specify 会为每个使用者分配独立的“个人网盘”，防止数据串栈。
*   *注：此区域在开发态为只读展示区，用于在编写指令时辅助理解路径结构，开发者无法在此直接上传文件。*
*   **标准目录结构**：
    *   📁 **`workspace/` (主工作区)**：AI 生成的最终产出物（如报告、代码）默认保存在此。
    *   📁 **`temp/` (临时区)**：存放中间推演结果或长篇草稿。系统定期清理，适合处理无需永久保存的数据。
    *   📁 **`memory/` (记忆磁盘)**：长效记忆存储区。保存使用者的个性化偏好或跨对话长效结论，实现“越用越懂你”。
    *   📁 **`assets/` (资产区)**：最终使用者在前端对话框中上传的文件（如图片、PDF）会自动转存至此，供 AI 读取。
    *   🔒 **`mailbox/` (信箱区)**：系统级目录，用于使用者间通信或系统消息投递（开发过程中一般无需关注）。

### 3.2 右侧上方核心画布（指令区）
一张没有任何多余边框的纯净白板。专注编写「系统提示词」。当你在左侧资源栏 Hover 文件时，可点击 `[📋]` 快速复制路径并粘贴到此处。

### 3.3 右侧下方能力面板（工具箱）
摒弃了传统的卡片网格平铺，采用 **“纵向详情大列表 (List View)”** 设计。为保护上方指令画布的体验，**下方能力面板设定固定最大高度（约占比 40%），内部出现垂直滚动条**，允许列表向下无限扩展。

采用这种排版的核心目的是为了清晰地展示每个工具的**“一句话描述”**以及**“内部嵌套依赖关系”**，避免因为卡片过小而导致信息截断。

能力严格分为三大区块，排序如下：

1.  **官方原子工具**：平台预设的基础能力（如读写文件、网络搜索）。
2.  **我的自建 Agent**：开发者自己创建的工具集合。数量无上限。列表末尾常驻 `[➕ 创建子 Agent]` 按钮。
3.  **接入的 MCP 服务**：开发者通过 SSE URL 等方式接入的外部底层系统能力。列表末尾常驻 `[➕ 接入私有 MCP]` 按钮。

**列表项结构说明：**
*   **左侧**：勾选框（启用/禁用状态）+ 图标 + 工具名称。
*   **中侧**：一句话工具描述（灰色字体，单行截断）。如果是被封装的 Agent，下方还会通过 `└─ 内部依赖:` 树状结构展示其底层调用了哪些原子工具。
*   **右侧**：对于可编辑的项（如自建 Agent 和 MCP），悬停或常驻显示 `[⚙️设置]` 入口。

**【重要微交互：选中即置顶】**
在任一区块内，当开发者勾选（`☑`）启用某个工具项后，**该项会高亮，并自动平滑过渡排序到该区块的最前面**。方便一眼看出当前 App 到底挂载了哪些实际生效的能力。

**【核心规则：基于底层平台的动态降级与屏蔽】**
由于不同大模型平台（如 Claude、GPT、DeepSeek）的底层能力差异极大，工具面板必须进行严格的**按需渲染（动态屏蔽机制）**：
1. **工具维度的屏蔽（整行隐藏）**：
   * 平台预设的基础工具和 Agent 会判断当前选择的大模型（顶部状态栏中间）是否具备对应能力。
   * **举例**：如果开发者在此页面顶部选择了 `DeepSeek`（纯文本模型，不支持多模态视觉），那么下方能力面板中所有强制依赖视觉能力的工具（如“图片审查”、“视频提取”）将**完全不显示**，避免开发者勾选后导致调用必败。
2. **配置维度的降级（内部参数置灰拦截）**：
   * 对于某些具有“弹性能力”的基础工具，工具本身会显示，但其内部某些高级配置项会根据平台能力被禁用。
   * **举例**：平台提供了一个 `Spread` (网页读取) 工具，它默认读取文字，但内置了一个 `[ ] 开启识图读取` 的高级开关。如果当前主脑是 `DeepSeek`，`Spread` 工具依然可以在列表中勾选使用，但在点击 `[⚙️设置]` 弹出的弹窗中，那个 `[ ] 开启识图读取` 的开关将被**置灰不可点**，同时旁边附带一个醒目的叹号图标。鼠标 Hover 时提示：`⚠️ 当前选用的平台模型不支持图片识别能力，该选项不可用。`
   * *(注：采用置灰而非直接隐藏的好处在于，让开发者知晓平台存在这个强大的能力，只是受限于当前选型的模型。这能起到教育用户和引导他们尝试更高级模型的作用。)*

## 四、 嵌套创建与配置：居中弹窗 (Modal)

为了保持主 App 上下文的连贯性，一切深层编辑收敛于居中弹窗。

### 4.1 触发与展现
*   **触发**：点击列表下方的 `[➕ 接入私有 MCP]`、`[➕ 创建子 Agent]`，或点击已配置列表项右侧的 `[⚙️设置]`。
*   **展现**：在屏幕正中央弹出一个宽大的**模态弹窗 (Modal)**。此时底层的主工作区变暗或轻微虚化，开发者视线高度聚焦于弹窗内的配置任务。

### 4.2 弹窗内容 (以创建子 Agent 为例)
弹窗内的布局是“主界面的微缩版”：
1.  **顶部标题**：输入/修改子 Agent 名称（必填，仅限英文下划线）和一句话描述。
2.  **指令设定**：填写子 Agent 专属的 Prompt。
3.  **能力分配**：下方列出可分配给该子 Agent 的工具选项。
4.  **底部操作**：`[取消]` | `[保存并关闭]`。

### 4.3 防循环引用的防呆设计
在配置子 Agent 的弹窗中，能力分配列表里 **所有可用的子 Agent 均正常显示**，但对于其勾选后可能造成循环引用的节点，**卡片状态置灰且不可勾选**（Hover 时可提示“存在循环引用，无法挂载”）。通过这种方式从根本上阻断套娃死锁的可能，同时保持能力列表的完整性。

## 五、 嵌套依赖透出与精准容错机制

在复杂的嵌套场景下（如：主 App 引用 Agent A，A 引用 B 和基础工具 c。当 c 失效时），采用**“透明依赖、精准定位、就地警告”**的柔性策略。

### 5.1 依赖关系的透明透出（微缩图标）
不再将子 Agent 视作黑盒。
*   **列表内嵌依赖**：在【我的自建 Agent】和带有封装特性的列表项下方，会通过树状图结构（`└─ 内部依赖:`）直接列出它内部封装了哪些基础能力。
*   **直观展示**：不再需要 Hover，这种结构在列表模式下默认可见，极大提升了开发者对复杂 Agent 的结构认知。

### 5.2 直观的就地异常标示 (Where it breaks, where it warns)
摒弃右上角全局的统一报警器，采用“哪错指哪”的直观设计。
当底层原子工具（如 c）发生异常（例如凭证失效、网络不通）时：
*   **源头直接警告**：在能力面板列表中，真实的异常源头（工具 c）整行会立刻呈现红色警告背景或边框，并打上醒目的 `[⚠️ 异常]` 标签。
*   **Hover 快速诊断**：开发者将鼠标悬停在该异常列表项上，会浮出具体的错误原因气泡，并在气泡内提供快捷修复按钮（如 `[去配置]`）。
*   **依赖链轻微透出**：如果某个自建的 Agent A 内部使用了失效的工具 c，那么 A 列表项下方 `└─ 内部依赖:` 中的工具 c 名称会变成红色。提醒开发者该组件“部分功能受损”，但 A 列表项整体不会大面积爆红，避免造成视觉恐慌。

### 5.3 运行时强拦截机制
*   **调试拦截**：仅当开发者点击顶部的 `[▶ 进入调试]` 时，系统会自动检查当前已勾选的能力链。如果发现有包含 `[⚠️ 异常]` 标签的列表项，系统会通过居中弹窗拦截：“当前勾选的工具 [XXX] 存在配置异常，请修复或取消勾选后重试”。给予开发者最终的控制权。
